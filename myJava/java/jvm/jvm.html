JVM - wolai 笔记

内存管理

JVM内存区域

程序计数器、虚拟机栈、本地方法栈、堆、元空间(在内存里面)
  • 元空间：本质和永久代类似，用来存放虚拟机加载的类型信息，常量静态变量。元空间在本地内存，永久代在JVM。
  • 虚拟机栈：(线程私有)每个线程有一个栈，存放的叫栈帧。每个方法执行的时候，都会创建一个栈帧。栈帧内存的是局部变量表、操作数栈、方法出口等信息。
  • 本地方法栈：与虚拟机栈类似，存储的是native方法
  • 程序计数器：(线程私有)行号计数器，记录的是代码执行到哪了。JVM里面唯一一个不会爆内存的区域。
  • 堆内存：堆是JVM所有线程共享的部分，在虚拟机启动的时候就已经创建，所有对象和数组都是在堆上进行分配。堆的物理空间可以是不连续的，但是逻辑上是连续的。这部分空间可以通过GC进行回收。jdk1.8之后字符串常量池从永久代剥离出来，放在堆中。
若一个方法的参数为空，那么局部变量表里面有没有变量？
  1. 若是静态方法，局部变量表里面没有任何方法
  2. 若是非静态方法，局部变量表里面有this
public class VarDemo1 {

    }

    }
}

JDK版本不同内存的变化

1.6使用永久代实现方法区
1.7将字符串常量池、静态变量放在了堆上
1.8在内存中划出了一块区域叫元空间，取代了永久代
因为永久代容易发生内存溢出问题，大小受JVM的限制
移到内存后，只要系统内存够用，就可以一直用

对象销毁过程

对象不再被任何引用指引时，就会变成垃圾，被垃圾收集器清除。
垃圾收集器(GC)通过标记清除、标记复制、标记整理等算法来回收内存，将对象占⽤的内存空间释放出来。

堆内存如何分配的

指针碰撞

分配内存时，虚拟机会维护一个指针，指向下一个可用的内存地址，每次分配内存时，只需要将指针后移一段距离，如果没有发生碰撞，就将这段内存分配给对象实例。

空闲列表

JVM维护一个列表，记录为占用的内存块。有新的对象请求内存时，JVM会遍历空闲列表，寻找足够大的空间来存放新对象。

堆和栈的区别？

  • 用途：栈用来存储局部变量、方法调用的参数、方法返回地址以及一些临时数据，方法被调用，就会创建栈帧。堆是用来存 new出来的对象。
  • 生命周期：栈内数据生命周期是当方法调用结束后对应的栈帧就会被销毁。堆是当垃圾回收机制(GC)检测到对象不再被引用的时候才被回收。
  • 存取速度：栈的存取速度快，对象在堆上的分配和回收需要更多的时间。
  • 存储空间：栈的空间小且固定，堆的空间大。
  • 可见性：栈是线程私有的。堆是线程共享的。

栈内存的是指针还是对象？

栈存的是基本类型的数据和对象的引用，比如说在方法中从新创建了一个对象MyObject obj = new MyObject();，栈帧存的obj是栈的引用，指向堆内的地址。

堆分为哪几部分？

  • 新生代：新生代分为伊甸园区(Eden Space)和幸存者区(Survivor Space)，新创建的对象放在伊甸园，该区满了的话执行垃圾回收机制，存活下来的对象移入生存者区。生存者区有两个，轮流充当对象的中转站。
  • 老年代：经理多次的垃圾回收机制还存活的对象，移入老年代区，该区对象存活时间长，执行垃圾回收的次数少。
  • 元空间：存储类的结构信息，如：字段、方法信息等。
  • 大对象区：用来存放需要占用大量内存的对象，一般直接放在老年区了。

String

字符串的值是不可变的，所有存储在字符串常量池内。
String s = new String（"abc"）执行过程
  1. 在堆上创建String对象S。
  2. 在字符串常量池寻找有没有这个字符串
    1. 有：直接将S指向这个位置。
    2. 无：创建一个字符串常量放在池内。
  3. 返回。

引用类型有哪些？

  1. 强引用：A a = new A(); 永远不会被GC回收
  2. 软引用：用SoftReference描述，发生内存溢出前会被回收。
  3. 弱引用：用WeakReference描述，下次GC来就回收了。
  4. 虚引用：也是GC来就被回收了，可用用来管理堆外内存。

弱引用的使用场景

  • 缓存系统：当希望缓存项目在内存压力下可用自动释放时，可以用弱引用来维护。
  • 对象池：用来管理那些暂时不用的对象
  • 避免内存泄漏：当一个对象不应该被长期引用时，使用弱引用可以防止对象被意外的保留造成内存泄漏。
使用弱引用的时候要注意，对象被垃圾回收后返回值会变成null，代码要记得检查这一点。

内存泄漏和内存溢出

内存泄漏

就是不再使用的对象仍被引用，导致垃圾回收机制没办法回收这些区域。
常见原因：
  • 静态集合： 静态变量的生命周期与应用程序相同，如果静态集合类（如HashMap、ArrayList等）保存了大量对象的引用，而这些对象实际上已经不再需要，就会导致内存泄漏。
  • 事件监听：未取消对事件源的监听，导致对象被持续引用。
  • 线程：未停止的线程可能持有对象的引用，无法回收、

内存溢出

在申请内存时，无法找到足够的内存 OutOfMemoryError
常见原因：
  • 大量对象的创建
  • 持久引用
  • 递归调用

内存溢出有哪几种情况？

  • 堆内存溢出OutOfMemoryError:Java heap space：创建了很多大对象，或者发生了内存泄漏。
  • 栈溢出StackOverFlowError：递归调用太深。
  • 元空间溢出OutOfMemoryError: Metaspace：系统的代码非常多或引用的第三方包非常多或者通过动态代码生成类加载等方法。
  • 直接内存溢出OutOfMemoryError: Direct buffer memory：

类的初始化和类加载

对象的创建过程

类加载检查：执行new之后，先检查之前是否被加载过，没加载过的先加载，然后再分配内存。
分配内存：不同的垃圾回收方法，适用于不同的分配方法
  1. 空闲指针：系统维护一个空闲指针，当空闲区域满足当前所需就成功分配，不成功就往后移。
  2. 空闲列表：从列表内找到大小合适的空闲区域。
不同线程的堆抢占：
当不同线程同时new对象的时候，可能会抢占位置。解决方法有两种：
  • 使用自旋锁
  • 每个线程有一个属于自己的TLAB，用于存放自己创建的对象。只有自己的TLAB放不下的时候，才会在堆中进行内存的分配。
对象头：对应哪个类的实例，哈希值，年龄带信息。

对象生命周期

  • 创建
  • 使用
  • 销毁

类加载器

用来把Java类加载到Java虚拟机中

  • 启动类加载器(Bootstrap class Loader):这是最顶层的类加载器，负责加载Java的核心库(如位于jre/lib/rtjar中的类)，它是用C++编写的，是M的一部分。启动类加载器无法被Java程序直接引用。
  • 扩展类加载器(Extension Class Loader):它是Java语言实现的，继承自ClassLoader类，负责加载Java扩展目录((jre/lib/ext或由系统变量Java.ext.dirs指定的目录)下的jar包和类库。扩展类加载器由启动类加载器加载，并且父加载器就是启动类加载器。
  • 系统类加载器(System Class Loader)/应用程序类加载器(Application Class Loader):这也是Java语言实现的，负责加载用户类路径(ClassPath)上的指定类库，是我们平时编写Java程序时默认使用的类加载器。系统类加载器的父加载器是扩展类加载器。它可以通过ClassLoader.getSystemClassLoader0方法获取到。
  • 自定义类加载器(Custom Class Loader):开发者可以根据需求定制类的加载方式，比如从网络加载class文件、数据库、甚至是加密的文件中加载类等。自定义类加载器可以用来扩展Java应用程序的灵活性和安全性，是ava动态性的一个重要体现。

双亲委派模型

双亲委派模型：当一个类收到加载委派请求的时候，先不会自己尝试加载这个类，而是交给父类，父类不行的话交给父类的父类，最终加载请求传送到顶层启动类加载器。当都无法加载这个类的时候，才会用当前所在的类加载器加载。
目的：确保类只被加载一次，避免重复加载。

双亲委派模型失效

Tomcat热部署，需要动态支持类的创建，所以双亲委派模型失效。

类加载过程

  • 加载：通过全限定名(包名+类名)，获取到该类的.class文件的二进制字节流。
  • 验证：确保被加载类的正确性，不会危害到虚拟机的安全。
  • 准备：为属性赋默认初始值
  • 解析：解析阶段是虚拟机将常量池的「符号引用」直接替换为「直接引用」的过程。
  • 初始化：运行构造方法

垃圾回收

是一种内存管理机制，负责释放不再被使用的对象所占的内存。
是自动执行的，避免了开发人员手动管理内存。

如何触发垃圾回收

  • 内存不足时：自动触发。
  • 手动请求：通过System.gc()调用。
  • JVM参数：调整JVM参数来调整垃圾回收行为。
  • 对象数量或内存使用达到阈值

不同场景下的垃圾回收

Minor GC、Major GC、Full GC

Minor GC

作用范围：在年轻代进行回收
触发条件：当Eden空间不足时，会执行该GC
特点：发生的频繁，回收效率高

Major GC

作用范围：主要针对老年代进行回收，但不一定只回收老年代。
触发条件：老年代空间不足，或者年轻代晋升速度过快时执行
特点：发生的频率较低，每次回收需要的时间长。

Full GC

作用范围：整个堆内存
触发条件：手动调用，年轻代升级时老年代空间不足，元空间空间不足。
特点：是昂贵的操作，因为要停止所有工作的线程STW(Stop The World)。和major gc的区别。

判断垃圾的方法

引用计数法

有一个变量记录当前类的引用数，多一个地方引用，数量加一。计数器为0时代表不被任何引用，可以被回收。
缺点：两个对象相互引用，但不被其他对象引用的时候，计数器不为0，不能被删除。

可达性分析

原理：从GC Root出发，向下追溯他们的引用对象，以及这些对象引用的其他对象。
如果从Root到某个对象不可达，那么这个就可以被回收。

回收算法有哪些

  • 标记-清除算法：通过可达性分析标记出所有不可达对象，然后统一回收所有被标记对象。缺点：效率不高、会造成大量碎片空间。
  • 复制算法：内存分为两部分，每次申请内存只是用其中一部分，内存不够时执行回收算法，然后把存活的复制进另一部分，当前部分清空。缺点：只有一半的内存被利用了。
  • 标记-整理算法：标记后，将存活对象都移到内存一端，然后直接清理剩余部分。
  • 分代回收算法：内存分为新生代和老年代，不同代的GC频率不同。

垃圾回收器有哪些？

CMS

标记-清除算法
以最小停顿时间为目标，适用于需要高响应的场景。

流程

  • 停止所有用户线程
  • 开始标记
  • 重新标记
  • 并发清除

优缺点

优点：响应时间快，停顿时间短
缺点：会导致内存碎片的积累

G1

标记-复制算法。
以控制GC停顿时间为目标，兼具高吞吐量和低延迟，适合大内存、多核环境。

适应场景

  • 大堆内存
  • 对内存碎片比较敏感
  • 比较平衡的性能：提供较低停顿时间的同时，也保持了较高的吞吐量。

G1特点

  1. 引入了分区思路，弱化了分代的概念。
  2. 合理利用垃圾收集器收集各个周期的资源
相对于CMS的改进：
  • 算法：标记-整理算法不会产生内存碎片
  • 停顿时间可控：G1可以设置预期停顿时间
  • 并发与并行：G1能充分利用多核环境下的硬件优势，缩短停顿时间。

CMS和G1的区别

  1. 使用范围：CMS是老年代收集器，G1是老年代和新生代。
  2. STW时间：CMS以最小停顿时间为目标，G1可以预测垃圾回收的停顿时间。
  3. 垃圾碎片：CMS使用标记-清除算法，碎片多。G1是标记-整理算法，没有内存碎片。
  4. 垃圾回收过程不一样：
    1. CMS：初始标记，并发标记，重新标记，并发清除。
    2. G1 ：初始标记，并发标记，最终标记，筛选回收。
  5. CMS会产生浮动垃圾

GC只作用在堆上吗

不仅会对堆进行垃圾回收，还会对方法区进行垃圾回收。
