计网 - wolai 笔记

常见

输入URL到页面展示

  1. 输入URL按下回车：浏览器解析协议、域名、路径等部分
  2. DNS解析：查找到对应的IP地址
  3. TCP连接：三步握手
  4. 客户端发送HTTP请求：请求行、请求头和请求体。
    1. 请求行：包含请求方法（如 GET、POST）、请求的 URL 路径和 HTTP 版本。
    2. 请求头：包括浏览器信息、语言类型、缓存控制等信息。
    3. 请求体:发送的数据
  5. 服务器处理请求：处理客户端的请求
  6. 服务器发送HTTP响应：响应行、响应头和响应体。
    1. 响应行指示处理结果（如 200 OK、404 Not Found），
    2. 响应头包含缓存控制、内容类型、日期等信息，
    3. 响应体则是实际的页面内容（HTML、CSS、JavaScript、图片等）
  7. 浏览器渲染页面：加载内容
  8. 关闭TCP连接：无通信的时候就关闭连接。

HTTP状态码

1XX 请求已被接受

100：收到请求的初始部分，应继续发送
101：服务器同意切换协议

2XX 成功响应

200：返回数据
201：已创建新资源
204：不返回任何内容

3XX 重定向响应

301：资源已被永久移动
302：资源被临时移动

4XX 错误

400：请求参数不正确
401：没有进行身份验证
403：服务器拒绝请求
404：资源不存在
405：请求方法类型不支持

5XX 服务器错误

500：服务器内部错误
502：无效响应
503：服务器暂不可用
504：服务器未接到上游服务器的响应

HTTP和HTTPS的区别

都是web网络传输协议，HTTPS是安全的
  • 安全性：HTTPS安全，因为加了SSL/TLS 加密层。
  • 端口号和前缀：http是80，https是443
  • 部署成本：https成本高，因为需要SSL/TLS证书

GET和POST区别

  • 传递数据方式：GET是URL传递参数，POST是请求头传递参数
  • 安全性：URL传递参数不安全，POST相对安全。
  • 使用场景：GET用于查询数据，POST用于修改数据
  • 请求缓存：GET会被请求缓存

TCP和UDP的区别

  • 连接方式不同：TCP需要三步握手连接，UDP无连接
  • 可靠性不同：TCP是可靠的，UDP是不可靠的
  • 拥塞控制：TCP有拥塞控制，确保网络拥塞时不会丢包。
  • 数据传输顺序：TCP是顺序到达的，即使乱了也会再排序。UDP可能会乱序到达。
  • 使用场景不同：TCP适用于高可靠的场景，如网页浏览。UDP适合实时性高的场景，可以丢包。
  • 服务对象不同：TCP是一对一，UDP可以一对多

TCP为什么可靠

校验和、序列号确认、流量控制、拥塞控制、超时重传

TCP流量控制

让发送方根据接收方的能力调整发送速率

OSI模型

应用层，传输层，网络层，数据链路层，物理层

常见端口

HTTP 80，HTTPS 443，DNS 53

应用层

请求类型

  • GET：获取数据用（可以数据缓存）
  • POST：提交数据用
  • PUT：更新数据用
  • DELETE：删除

HTTP长连接

HTTP连接是基于TCP连接实现的，建立HTTP连接之前需要建立TCP连接
这样的话实现的是HTTP短连接（左图），因为每次HTTP请求的时候都要进行TCP连接。
长连接是HTTP响应结束后，TCP也不断开连接（右图）。
只要没有一方提出断开连接，则保持连接状态。

HTTP不安全

由于是明文传输的
  • 窃听风险
  • 篡改风险
  • 冒充风险
HTTPS改进
  • 信息加密
  • 校验机制
  • 身份证书

HTTP、SOCKET和TCP的区别

  • HTTP：超文本传输的应用层协议
  • Socket：套接字，是计算机网络中的一种抽象
  • TCP：面向连接的可靠的传输层协议

cookie和session

  • cookie存储在客户端，session存储在服务器端
  • cookie有大小限制，session没有
  • cookie不安全，session安全
  • cookie因为数据随每个请求发送到服务器，可能会影响网络传输效率，尤其是在Cookie数据较大时。使用Session时，因为数据存储在服务器端，每次请求都需要查询服务器上的Session数据，这可能会增加服务器的负载，特别是在高并发场景下。
禁用cookie后，session通常也无法使用，因为需要使用cookie传送会话ID。
解决办法？就是把会话ID传送给服务器，URL重写、隐藏表单。

local Storage和Cookie

  • local Storage的容量大
  • cookie每个请求都发送给服务器，local Storage仅在本地使用
  • cookie有生命周期，localStorage没有
  • local Storage适合存放不同页面之间共享的数据

token

JWT就是token的一种，存储用户的信息，服务器端拿到token后需要进行解密。

JWT

安全，无状态，可跨域

字段

头部，负载，签名

但是在集群部署中，服务器端不会共享这样的数据，导致用户在不同服务器之间切换时需要重新登录。

RPC

用处：希望程序员能像调用本地方法那样去调用远端的服务方法
http主要是bs架构，RPC是cs架构用得多
一般企业内部使用RPC，对外使用http

Nginx

负载均衡算法
  1. 轮询：依次使用不同服务器
  2. IP哈希：根据客户端ip进行hash计算使用哪个服务器
  3. URL哈希
  4. 最短响应时间：哪个快用哪个

传输层

三步握手

为什么三步

  1. 阻止重复历史连接的初始化：客户端得到的ack不是他想要的，就会拒绝连接
  2. 同步双方的初始序列号
  3. 避免资源的浪费

第一步丢包

客户端继续重发syn，直到超过设定的重发最大次数
每次重发的时间间隔会加倍，比如1s,2s,4s,8s,16s,32s

第二步丢包

客户端没收到ACK，会进行重发
服务器端发力SYN，没收到回复，也会重发

第三步丢包

服务器端发送syn之后，客户端进行第三次ack回应，客户端进入ESTABLISH 状态，但是这个包丢了。
那么服务器过一会就会重新发送syn包，直至收到客户端ack或者发送次数超过最大限制。
因为ack包不会重传

握手队列

三步握手完成后，会放入全连接队列，调用 accpet 就是从把连接取出来给用户程序使用。
第一步握手完成后会放入半连接队列
当有大量的SYN包到达时，队列会满，这样就会丢弃新收到的连接。

四步挥手

客户端挥手，进入FIN_WAIT状态，服务的ack之后进入CLOSE_WAIT状态
服务器端发送FIN之后进入LAST_ACK，客户端进入TIME_WAIT 阶段
当服务器端没数据要发送的时候，第二步和第三步也会和为一步。

客户端FIN丢包

会进行重发，但达到重发次数后，就会自动进入close状态。

为什么等2MSL

网络中可能存在来自发送方的数据包，当这些发送方的数据包被接收方处理后又会向对方发送响应，所以一来一回需要等待 2 倍的时间。

服务端出现大量的timewait有哪些原因

出现timewait证明服务器端是主动断开的一方
问题来了，什么场景下服务端会主动断开连接呢？
  1. HTTP 没有使用长连接
  2. HTTP 长连接超时
  3. HTTP 长连接的请求数量达到上限

TCP粘包

  • 粘包：发送方发送的多个数据包被接收方合并为一个数据包接收。
  • 拆包：发送方发送的一个完整的数据包被接收方拆分成多个数据包接收。
解决办法：固定长度、特殊分隔符

拥塞控制

当网络发生拥堵时减少数据包的发送量，来缓解拥堵。
通过拥塞窗口控制发送数据包数量的多少，没拥塞就增大，有拥塞就缩小。
当没有在规定时间内收到ACK，就认为是发生了拥塞。
慢启动
开始的时候发1个包，收到ACK之后发2个，4个，8个，指数增加。
拥塞避免
慢启动达到阈值之后，开始一次增加1个，线性增加
拥塞发生
阈值变为当前滑动窗口大小的一半，滑动窗口变为1，继续慢启动(不是最优)
快速恢复
给发送方发送三个重复的ACK
阈值设置为滑动窗口值的一半，滑动窗口从阈值开始线性增加


保活计时器

当客户端和服务器连接成功后，客户端突然故障。不能让服务器一直等下去，而要判断客户端是否存活。
每收到客户端发来的数据后，保活计时器就会重新开始计时。
计时结束还没有收到，服务器就会发送探测报文，75秒发送一个，如果10个之后还没有收到，那么就关闭这个连接。
